AI 協作提示詞紀錄：產品演進與 Prompt 設計歷史

本文件完整記錄了「機器學習十大演算法全互動實驗室」從一個簡單的靜態研讀手冊概念，到演進為一個具備前後端分離、互動 Canvas、隨堂測驗、雙引擎 AI 代理的成熟 Web App 產品協作歷程。

🎯 第一階段：高保真 UI 還原與免金鑰 RAG 概念

1. 協作目標

使用者上傳了手繪/設計原型圖（包含左側單元指南、中間 Canvas 數據實驗、右側 AI 助理聊天室、下方快捷問答標籤），希望將靜態 ML 介紹網頁進行大規模優化，並實作一個不需要 API Key 也能提供精準回答的智慧功能。

2. 提示詞設計邏輯 (Prompt)

引導思路：如何做到免金鑰問答？如果是純靜態網頁，最好的方式是「前端語義關鍵字匹配 (Local Semantic Match)」。我們需要建立一個結構化的 Knowledge Base，並透過前端 JavaScript 進行 token 匹配。

UI 建構提示詞：

「請為我優化這個機器學習演算法介紹網頁。我希望還原設計圖的 Premium 科技感深色調佈局。左側為十強演算法手冊速查欄，右側為對話視窗。下方有橫向滾動的常見問題快捷 Chip 按鈕。在沒有金鑰的情況下，系統能利用本地 RAG（關鍵字特徵匹配算法）在報告文本中進行特徵評分，找出最吻合的章節，並呈現圖文並茂（含 LaTeX 數學公式）的專業解答。」

🐞 第二階段：修復技術痛點 (LaTeX 與 KaTeX 加載衝突)

1. 遇到的 Bug 痛點

在執行時，系統拋出 No element provided to render 錯誤，且 LaTeX 數學公式中（例如線性迴歸 MSE 公式、SVM 超平面公式）的反斜線 \ 導致 KaTeX 發生解析錯誤崩潰。

2. 技術診斷與解決提示

診斷一：KaTeX 的自動渲染腳本放於 <head> 並利用行內 onload 觸發，但此時 DOM 的 document.body 尚未渲染完成，傳入了 null，導致崩潰。

診斷二：在 JavaScript 的字串樣板中，單反斜線 \ 是轉義字元。\frac 會被誤解析為 Form Feed 控制字元 \f。

修正提示詞與解法：

「1. 請將 KaTeX 自動渲染 renderMathInElement 從 head script onload 中移除，改在最下方的 DOMContentLoaded 事件（確保 DOM 樹安全就緒）中調用。\n2. 對所有的 LaTeX 公式進行轉義，將單反斜線 \ 統一替換為 JavaScript 轉義安全的雙反斜線 \\（例如 \\frac、\\sum）。」

🔄 第三階段：版面微調與滾輪鎖死限制移除

1. 遇到的 Bug 痛點

原本版面設計為了配合彈窗，在最外層的 body 設定了 overflow-hidden 且配合了複雜的卡片嵌套高度，導致在筆電或小螢幕上「互動實驗室」和「研讀介紹」卡片被截斷，無法用滾輪下滑檢視。

2. 修正提示詞與解法

優化思路：將原先分散的多重卡片容器解耦，並將中央工作區（center-pane）重新設計為一個單一且高度流暢的滾動視窗。

修正提示詞：

「版面怪怪的，不能用滾輪下滑導致有些視窗看不到。請幫我優化 body 的 height 設定，將 min-h-screen 鎖定為 h-screen 貼合瀏覽器。將中央工作區解耦，讓『數據實驗卡片』與下方『研讀介紹卡片』融合成一個整體的流暢滾動區域，當滑鼠滾輪往下滑動時，能像閱讀精美文章一樣極其滑順地檢視完整報告，且內部不會再有滾輪焦點衝突。」

🏆 第四階段：功能大躍進 (前後端解耦、測驗系統、評析矩陣)

1. 協作目標

為了讓平台更像一個標準的產品，使用者提出需要：

前後端分離：將海量資料抽離，避免 index.html 肥大。

自我學習測驗：針對各單元動態生成隨堂選擇題，包含即時答題回饋與深層背後原理、數學邏輯解析。

評析解說：擴充所有演算法的「優點、缺點限制、真實應用場景」結構化表格。

後端部署：Flask 伺服器代理呼叫 AI 大模型。

2. 架構演進提示詞與實作

Python 後端代理設計：利用 Flask 建立 /api/chat。當前端提問時，會將請求發送至後端，後端自動附加安全環境變數中的 GEMINI_API_KEY，轉發給 Gemini/OpenAI 伺服器。這徹底隱藏了金鑰，並完美解決了瀏覽器直連時最常遇到的 CORS 跨網域限制。

解耦資料庫提示詞：

「請為我進行前後端部署與優化：\n1. 撰寫一個 app.py（Flask 後端），提供靜態路由與安全 AI Chat Proxy，讀取 .env 的金鑰並發送 requests，杜絕金鑰外洩與 CORS 限制。\n2. 將 11 大單元的所有 Markdown、LaTeX、隨堂測驗、優缺點評析表格，抽離成獨立的 data.json。\n3. 前端 index.html 在初始化時透過 Fetch API 動態載入 data.json，並新增一個隨堂測驗面板，當切換至測驗時，顯示對應演算法的挑戰題，並提供精緻的答對/答錯與原理詳盡解析視覺回饋。」

💡 提示詞協作心得總結

這個專案之所以能取得如此精緻的高品質產出，在於我們遵循了以下協作模式：

漸進式架構演進：不急於一開始就把所有東西塞進單一檔案，而是先確立 UI，再解 Bug，最後進行前後端解耦。

LaTeX 特殊轉義警覺：在涉及 Canvas 數學運算與前端 LaTeX 渲染時，嚴格使用 \\ 進行字串級轉義。

防禦性 Null 指標設計：在不同載入階段或響應式版面切換時，對所有的 DOM 元素（如 status-banner、toggle-btn）進行強固的 if (el) 存在檢查，確保產品具備工業級的穩定性。
